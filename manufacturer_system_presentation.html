02 / Module

Products Module

How the product page works from the user side and from the PHP side.

User Side
The form accepts the product name and quantity on hand. A table below shows all current records, with action buttons for editing or deleting each row.
Program Side
The PHP file checks which action was requested, then runs the matching SQL statement for insert, update, or delete. The page re-reads the table so the latest data appears immediately.
Why It Matters
Bound parameters keep SQL clean and safely pass input into Oracle. The updated table delivers instant feedback after every action.

03 / Modules

Components &
Suppliers Modules

Different data, same clean workflow and user experience.

⚙️
Components
Parts & materials used in manufacturing
  • Stores parts or materials used in manufacturing.
  • Each record includes a component name and description.
  • Full add, edit, delete, and view support — identical to Products.
🏭
Suppliers
Where materials & components come from
  • Stores supplier name, contact information, and address.
  • Helps track origin of all materials and components.
  • Uses the same structure — users need no relearning.
Both modules reuse the same navigation, form layout, and table style — improving usability and making the project significantly easier to maintain.

Oracle XE
Oracle 21c XE stores the main PRODUCT, COMPONENT, and SUPPLIER tables with identity-based primary keys.
PRODUCT_COMPONENT and SUPPLIER_COMPONENT model relationships instead of keeping records isolated.
Design is ready for future features: part tracing, material computation, supplier link views.
Sample data in the SQL file makes the system easier to test and demonstrate.
A simple but functional CRUD system that combines web pages, PHP logic, and a relational Oracle database.
